# 분석 보고서 파일 작성

현상 분석, 장애 조사, 시스템 점검 등의 결과를 **반드시 md 파일로** 작성한다.
채팅 응답으로만 보고서를 제공하는 것은 금지 — 재부팅/세션 종료 시 기록이 사라진다.

## 트리거

- "보고서", "report", "리포트", "기록해", "분석 기록", "보고서 작성"
- "조사", "전수조사", "원인 분석", "로그 분석", "장애 분석", "계획 없는 즉석 분석 요청"

## 적용 범위

**이 스킬은 계획서(프로젝트의 활성 plan 경로 — `_path-rules.md` 동적 폴백)가 없는 비정규 분석 작업에만 사용한다.**

| 상황 | 사용 스킬 |
|------|----------|
| 계획서가 있는 구현/수정 작업 완료 | `/done` (기존 계획서 체크·아카이브) |
| 계획서 없이 수행한 장애 조사·시스템 점검·원인 분석 | **`/report`** (이 스킬) |

**판단 기준**: 현재 대화에서 활성 plan 경로(`_path-rules.md` 동적 폴백 — 보통 `docs/plan/` 또는 `.worktrees/plans/docs/plan/`) 하위 계획서를 참조하여 작업했다면 `/done`이 맞다.
계획 없이 사용자의 즉석 요청으로 조사·분석한 경우 이 스킬을 사용한다.

## 핵심 규칙

1. **파일 우선**: 분석 결과는 반드시 `docs/report/YYYY-MM-DD_<slug>.md`로 저장
2. **채팅 금지**: 보고서 내용을 채팅으로만 출력하고 파일을 안 만드는 것은 금지
3. **중복 방지**: 같은 날짜+주제 파일이 존재하면 `-2`, `-3` 접미사 추가
4. **커밋 별도**: 파일 작성 후 자동 커밋하지 않음. 경로만 알리고 커밋은 사용자 판단
5. **plans 워크트리 분리**: 이 스킬은 보고서 전용이므로 plans 문서 커밋 경로가 아니며, plans 문서 수정이 함께 있으면 `/done` 또는 plans-aware 커밋 규칙을 따른다.
6. **조사 단계 수정 금지**: 조사 단계에서는 코드 수정·DB 변경·재시작을 하지 않는다. 수정이 필요하면 별도 승인 후 `/implement` 또는 일반 수정 흐름으로 전환한다.

## 실행 단계

### 1단계: 컨텍스트 수집

현재 대화에서 수행한 분석 내용을 정리:
- 조사 대상 (시스템, 기간, 범위)
- 확인한 증거 (로그, 이벤트, 코드, 덤프 등)
- 도출한 결론
- 권장 조치

### 2단계: 보고서 디렉토리 확인

```
docs/report/
```

디렉토리가 없으면 생성한다.

### 3단계: 파일 작성

**경로 규칙**: `docs/report/YYYY-MM-DD_<slug>.md`
- slug: 분석 주제를 kebab-case로 (예: `reboot-analysis`, `memory-leak-investigation`, `api-timeout-diagnosis`)
- 날짜: 분석 수행일 기준

**보고서 템플릿**:

```markdown
# <제목>

- **분석일**: YYYY-MM-DD
- **분석 대상**: 기간/범위/시스템
- **환경**: OS, RAM, 주요 H/W 등

---

## 1. 현상

무엇이 발생했는지. 사용자가 보고한 증상, 발생 시각, 빈도.

## 2. 조사 과정

어떤 로그/이벤트/코드/덤프를 확인했는지.
각 단계에서 발견한 사실을 시간순 또는 논리순으로 기술.

## 3. 원인 분석

근본 원인과 증거. 확정/추정 구분.
표나 타임라인으로 정리하면 좋음.

## 4. 영향 범위

어떤 서비스/기능/데이터에 영향이 있었는지.

## 5. 조치 사항

### 즉시 조치

실행 가능한 명령어는 코드블록으로 제공.

```powershell
# 예시
mdsched.exe
```

### 후속 조치

- [ ] 항목별 체크리스트

## 6. 재발 방지

모니터링/알림 설정, 코드 수정, 설정 변경 등.
```

### 4단계: 완료 알림

파일 경로를 사용자에게 알린다:

```
보고서 작성 완료: docs/report/YYYY-MM-DD_<slug>.md
```

## 보고서 유형별 가이드

### 장애/재부팅 분석
- 이벤트 로그 (Event ID, Provider, 타임라인) 필수 포함
- 프로세스별 메모리 사용량 표
- BSOD인 경우 BugCheck 코드 + 덤프 경로

### 성능 분석
- 측정 지표 (응답시간, 메모리, CPU)
- Before/After 비교표

### 버그 분석
- 재현 경로
- 스택 트레이스 또는 에러 로그
- 수정 내용 + 검증 결과

## 주의사항

- 보고서에 비밀번호, API 키, 개인정보 포함 금지
- 로그 원문은 핵심 부분만 발췌 (전체 복사 금지)
- 인코딩 깨진 한글은 의미를 해석하여 정상 한글로 기술
