# 디자인 프롬프트 생성

구현 AI가 자율적으로 UI를 설계·구현할 수 있도록, **사용자가 보고 조작하는 것**만 선언적으로 기술하는 문서를 생성한다.

## 트리거

- "디자인 프롬프트", "design prompt", "/design-prompt"

## 작성 원칙

### 선언적 기술
- **적는다**: "plan 목록이 보인다 (제목, 상태, 진행률)"
- **안 적는다**: "좌측에 카드형 패널로 배치한다"

### 금지 항목
- 위치 지정 (상단/좌측/우측/하단 등)
- UI 패턴 지정 (탭/필터/모달/드로어/카드/패널/그리드 등)
- 구현 기술 언급 (프레임워크, 라이브러리, API 스키마, 파일 경로 등)
- "기존", "현재", "사용자가~" 같은 현상 설명

### 서술 형태
- "X가 보인다", "X를 할 수 있다", "X 조건에서 노출된다" — 이 세 형태만 사용

### 충분한 기능 기술
- 각 기능 섹션에서 **보이는 정보**(어떤 필드/속성이 표시되는지)와 **가능한 조작**(버튼/토글/입력)을 빠짐없이 열거한다
- 한 줄 "X가 보인다"로 끝내지 않고, 그 안에 포함되는 세부 항목을 나열한다
- 예: "각 항목에 파일명, 상태, 브랜치명, 타임스탬프가 표시된다"

## 문서 구조

```markdown
# {페이지명} — Design Prompt

## 페이지 탭 (있으면)
(이 페이지가 속한 형제 탭)

---

## {페이지명} 요구사항

### 원칙
(모바일 우선, 최소 높이 등 전역 제약)

### {기능 A}
- X가 보인다
- Y를 할 수 있다
- Z 조건에서 노출된다

### {기능 B}
...
```

## 실행 단계

### 1단계: 요구사항 수집
사용자에게 확인: 대상 페이지, 원하는 변경

### 2단계: 디자인 프롬프트 작성

체크리스트:
- [ ] 모든 항목이 "X가 보인다" 또는 "X를 할 수 있다" 형태인가?
- [ ] 위치를 지정하고 있지 않은가?
- [ ] UI 패턴을 강제하고 있지 않은가?
- [ ] 구현 기술을 언급하고 있지 않은가?
- [ ] "기존", "현재" 표현이 없는가?

### 3단계: 저장
파일 위치: 대상 프로젝트의 `docs/` 하위
파일명: `design-prompt-{주제}.md`
