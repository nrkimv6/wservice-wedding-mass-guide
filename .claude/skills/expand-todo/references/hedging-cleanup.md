# Hedging 자기 검토

plan 본문(헤더·요약·구현 순서·기술적 고려사항 전체)을 다시 읽고, **방금 LLM이 직접 작성한 글에 자기 의심 톤이 남아있는지** 판단한다.

규칙은 한 줄이다:

> **"방금 쓴 글에 자기 의심을 남겼다면, 검증으로 해소하라."**

패턴 매칭/정규식/키워드 리스트로 단어를 찾지 않는다. 자기 글의 _톤_을 본다.

## 자문 질문

본문을 다시 읽고 다음 세 질문을 던진다:

1. **불확실 표현이 있는가?**
   "확인 필요", "(확인 필요)", "line ?", "재확인", "주의", "~인 듯", "~할 수도", "~인지 확인" 같은 톤.
   - 그 자리에서 직접 검증할 수 있었는가?
     - **YES** → Glob/Grep/Read로 즉시 검증해 사실로 교체. hedging 제거.
     - **NO** → 명시적 TODO 체크박스로 변환 (예: `Phase X에서 ~ 확정`).

2. **자기 의견 hedging이 있는가?**
   "~를 권장", "~가 안전", "~가 좋을 수 있다", "~방식이 권장된다".
   - 근거가 본문에 있는가?
     - **YES** → 단정문으로 교체.
     - **NO** → 삭제.

3. **자기 부정/유보가 있는가?**
   "이 분석이 부족할 수 있다", "더 조사 필요", "추가 검토 필요".
   - 부족하면 **지금 조사**한다. 또는 명시적 follow-up TODO 체크박스로 변환.

## 보존 (제거 금지)

- Phase R(재발 경로 분석), 검증 Phase의 "재검증" — 작업 자체가 검증인 경우
- ⚠️/🔴 위험 마커, 사용자가 명시한 우려점 인용
- `feedback_*.md`, `.claude/rules/`에서 강제하는 메모리 규칙 인용
- 사용자 결정이 필요한 사항 (LLM이 판단 불가능한 trade-off)
